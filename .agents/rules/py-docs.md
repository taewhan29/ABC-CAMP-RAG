# Python Documentation Rule

## 목적

이 규칙은 Python 소스 코드를 작성하거나 수정할 때 일관성 있고 유지보수 가능한 문서화를 적용하기 위한 규칙이다.

코드의 동작을 그대로 반복해서 설명하지 말고, 사용 목적, 입력과 출력, 제약 조건, 예외, 부작용을 중심으로 문서화한다.

## 기본 원칙

* 모든 공개 모듈, 클래스, 함수, 메서드에는 docstring을 작성한다.
* 내부 구현이 명확한 단순한 private 함수에는 불필요한 docstring을 작성하지 않는다.
* 코드만으로 명확한 내용을 주석으로 반복하지 않는다.
* 주석은 코드가 무엇을 하는지보다 왜 그렇게 구현했는지를 설명한다.
* 문서와 실제 코드가 다르면 코드를 기준으로 문서를 즉시 수정한다.
* 구현되지 않은 기능이나 보장되지 않은 동작을 문서에 작성하지 않는다.
* 문서화를 위해 함수의 실제 동작을 임의로 변경하지 않는다.
* 공개 API의 기존 의미와 호환성을 유지한다.

## 언어

* docstring과 코드 주석은 프로젝트에서 기존에 사용 중인 언어를 따른다.
* 기존 기준이 없다면 docstring은 한국어로 작성한다.
* 클래스명, 함수명, 매개변수명, 라이브러리명과 같은 기술 식별자는 원문을 유지한다.
* 번역투나 과장된 표현을 피하고 간결하고 명확하게 작성한다.

## Docstring 형식

Google Style Docstring을 기본 형식으로 사용한다.

```python
def calculate_total_price(
    unit_price: float,
    quantity: int,
    discount_rate: float = 0.0,
) -> float:
    """할인을 적용한 총 결제 금액을 계산한다.

    Args:
        unit_price: 상품 한 개의 가격.
        quantity: 구매 수량.
        discount_rate: 0 이상 1 이하의 할인율.

    Returns:
        할인이 적용된 총 결제 금액.

    Raises:
        ValueError: 가격이나 수량이 음수이거나 할인율이 유효 범위를 벗어난 경우.
    """
```

## 모듈 문서화

공개 기능을 제공하는 모듈에는 파일 최상단에 모듈 docstring을 작성한다.

모듈 docstring에는 다음 내용을 필요한 범위에서 포함한다.

* 모듈의 책임
* 주요 클래스와 함수
* 외부 시스템 또는 데이터와의 관계
* 사용 시 주의 사항

```python
"""사용자 계정 인증과 토큰 발급 기능을 제공한다.

이 모듈은 사용자 자격 증명을 검증하고 액세스 토큰을 생성한다.
데이터베이스 접근은 UserRepository를 통해 수행한다.
"""
```

단순 설정 파일이나 내용이 자명한 모듈에는 장황한 설명을 작성하지 않는다.

## 함수와 메서드 문서화

함수 docstring에는 필요한 경우 다음 항목을 작성한다.

* 한 문장으로 된 기능 요약
* 추가 동작 설명
* `Args`
* `Returns`
* `Yields`
* `Raises`
* `Example` 또는 `Examples`
* `Note`

다음 기준을 따른다.

* 요약문은 명령형보다 현재형 서술로 작성한다.
* 매개변수 설명에 타입을 반복하지 않는다. 타입은 type hint로 표현한다.
* 기본값은 시그니처만으로 충분하지 않을 때만 설명한다.
* `None`을 반환하는 함수에는 특별한 의미가 없다면 `Returns`를 생략한다.
* 실제로 발생시키거나 의도적으로 전달하는 예외만 `Raises`에 작성한다.
* 호출자가 알아야 하는 부작용이 있으면 명시한다.
* 파일 생성, 데이터베이스 변경, 네트워크 요청, 캐시 갱신 여부를 필요한 경우 설명한다.

## 클래스 문서화

클래스 docstring에는 클래스의 책임과 사용 방식을 설명한다.

필요한 경우 다음 내용을 포함한다.

* 클래스가 표현하는 개념
* 주요 상태
* 생성 시 요구 조건
* 수명 주기
* 스레드 안전성 또는 비동기 사용 조건

생성자 매개변수 설명은 클래스 docstring 또는 `__init__` docstring 중 한 곳에만 작성하여 중복을 피한다.

```python
class DocumentIndexer:
    """문서를 분할하고 임베딩하여 검색 인덱스에 저장한다.

    Args:
        embedding_model: 문서 임베딩 생성에 사용할 모델.
        vector_store: 임베딩과 메타데이터를 저장할 벡터 저장소.
        chunk_size: 문서를 나눌 최대 문자 수.
    """
```

## 속성과 데이터 모델 문서화

`dataclass`, Pydantic 모델 및 공개 데이터 객체는 각 필드의 의미를 명확히 설명한다.

필드명이 의미를 충분히 전달하지 못하거나 값의 단위, 범위, 형식에 제약이 있으면 반드시 문서화한다.

```python
@dataclass
class SensorReading:
    """센서에서 수집한 단일 측정값."""

    sensor_id: str
    """센서를 식별하는 고유 ID."""

    temperature_celsius: float
    """섭씨 단위의 측정 온도."""

    measured_at: datetime
    """UTC 기준 측정 시각."""
```

## 타입 힌트

* 공개 함수와 메서드에는 매개변수 및 반환값 타입을 작성한다.
* Python 내장 제네릭 표현을 우선 사용한다.
* 의미가 다른 값에 동일한 원시 타입을 반복해서 사용하면 `TypeAlias`, `NewType`, `Enum` 또는 데이터 클래스를 검토한다.
* 복잡한 반환 구조는 `dict`나 `tuple` 설명에 의존하지 말고 명시적인 타입으로 표현한다.
* `Any`는 외부 라이브러리 제약 등 불가피한 경우에만 사용한다.
* `Optional[T]` 또는 `T | None`은 실제로 `None`이 허용될 때만 사용한다.
* 타입 힌트와 docstring의 설명이 충돌하지 않도록 한다.

## 예제 코드

사용 방법이 직관적이지 않은 공개 API에는 짧은 실행 예제를 제공한다.

예제는 다음 조건을 충족해야 한다.

* 실제 공개 API를 사용한다.
* 불필요한 설정을 제외한다.
* 복사해 실행할 수 있는 형태를 우선한다.
* 존재하지 않는 클래스, 함수, 매개변수를 만들지 않는다.
* 출력값이 중요한 경우 예상 결과를 함께 보여준다.
* API 키와 개인정보 등 민감한 값을 포함하지 않는다.

```python
def normalize_score(score: float, maximum: float) -> float:
    """점수를 0과 1 사이의 값으로 정규화한다.

    Examples:
        >>> normalize_score(75, 100)
        0.75
    """
```

## 인라인 주석

인라인 주석은 다음 상황에서만 작성한다.

* 비즈니스 규칙의 이유를 설명해야 하는 경우
* 성능 또는 호환성을 위한 비직관적인 구현인 경우
* 외부 API의 제약이나 우회 방법을 설명해야 하는 경우
* 알고리즘의 중요한 단계나 수학적 근거를 설명해야 하는 경우
* 삭제하면 안 되는 코드의 배경을 설명해야 하는 경우

다음과 같은 주석은 작성하지 않는다.

```python
# 사용자 목록을 순회한다.
for user in users:
    ...
```

대신 의사결정의 이유를 설명한다.

```python
# 비활성 계정은 과금 집계 대상에서 제외한다.
active_users = [user for user in users if user.is_active]
```

## TODO와 FIXME

* `TODO`에는 남은 작업과 필요한 이유를 구체적으로 작성한다.
* 가능하면 이슈 번호나 추적 가능한 참조를 포함한다.
* 담당자 이름만 적거나 의미가 불분명한 TODO를 작성하지 않는다.
* 이미 해결된 TODO와 FIXME는 제거한다.

```python
# TODO(#142): 공급사 API가 페이지네이션을 지원하면 전체 조회 방식으로 변경한다.
```

## 비동기 코드

비동기 함수는 다음 내용을 필요한 경우 문서화한다.

* 호출 시 `await`가 필요한지
* 병렬 호출 가능 여부
* 취소 시 동작
* 타임아웃
* 재시도 정책
* 네트워크 또는 파일 I/O 수행 여부
* 동시성 제한

비동기 함수라는 사실만 반복해서 설명하지 않는다.

## 제너레이터와 컨텍스트 매니저

* 제너레이터는 `Returns` 대신 `Yields`를 사용한다.
* 중간에 발생할 수 있는 예외와 종료 조건을 설명한다.
* 컨텍스트 매니저는 진입 시 획득하는 자원과 종료 시 정리하는 자원을 설명한다.

## 예외 문서화

* 예외를 새로 발생시키는 경우 메시지는 문제 원인과 잘못된 값을 파악할 수 있게 작성한다.
* docstring에는 호출자가 처리할 가능성이 있는 예외를 우선 기록한다.
* 내부 구현에서 우연히 발생할 수 있는 모든 예외를 나열하지 않는다.
* 예외를 다른 예외로 변환하는 경우 원래 예외를 연결한다.

```python
try:
    value = int(raw_value)
except ValueError as error:
    raise InvalidConfigurationError(
        f"유효하지 않은 정수 설정값입니다: {raw_value!r}"
    ) from error
```

## README 및 별도 문서와의 정합성

Python 공개 API를 변경할 때 다음 파일이 존재하면 함께 확인한다.

* `README.md`
* `docs/**/*.md`
* `docs/**/*.rst`
* `examples/**/*.py`
* 튜토리얼과 샘플 코드
* API 레퍼런스
* 변경 이력

함수명, 매개변수, 반환 형식 또는 사용 절차가 변경되었다면 관련 문서와 예제를 함께 수정한다.

단, 현재 Glob 범위 밖의 파일을 수정해야 한다면 먼저 변경 필요성을 알리고 작업 범위를 명시한다.

## 문서 생성 도구

프로젝트에 Sphinx, MkDocs, pdoc 또는 다른 문서화 도구가 설정되어 있으면 기존 도구와 구조를 유지한다.

다음 파일을 우선 확인한다.

* `pyproject.toml`
* `mkdocs.yml`
* `docs/conf.py`
* `docs/index.md`
* `docs/index.rst`

새로운 문서화 도구를 임의로 도입하지 않는다. 도구가 설정되어 있지 않다면 먼저 가장 단순한 문서화 방식으로 개선한다.

## 검증

문서를 변경한 뒤 가능한 범위에서 다음을 확인한다.

1. docstring이 실제 함수 시그니처와 일치하는지 확인한다.
2. 매개변수명, 반환값, 예외 설명을 확인한다.
3. 예제 코드가 현재 API와 일치하는지 확인한다.
4. 프로젝트에 설정된 formatter, linter, type checker 및 테스트를 실행한다.
5. 문서 빌드 설정이 있다면 문서 빌드를 실행한다.
6. 실행하지 못한 검증 항목은 완료한 것처럼 표현하지 않고 명시한다.

프로젝트에 설정이 존재하면 다음 명령을 우선 사용한다.

```bash
ruff check .
ruff format --check .
mypy .
pytest
mkdocs build
sphinx-build -W docs docs/_build
```

설정 여부를 확인하지 않고 모든 도구를 설치하거나 실행하지 않는다.

## 기존 코드 수정 원칙

* 문서화 요청만 받은 경우 대규모 리팩터링을 수행하지 않는다.
* 코드 변경이 필요하면 문서 정확성을 위해 필요한 최소 범위로 제한한다.
* 공개 API 이름을 임의로 변경하지 않는다.
* 기존 docstring 형식이 프로젝트 전반에서 일관되게 사용되고 있다면 해당 형식을 우선한다.
* 자동 생성된 파일이나 외부 라이브러리 코드는 수정하지 않는다.
* 테스트 코드에는 테스트 목적이 명확하지 않은 경우에만 간단한 설명을 추가한다.

## 작업 결과 보고

문서화 작업을 완료하면 다음 내용을 간결하게 보고한다.

* 문서화한 모듈, 클래스 및 함수
* 추가하거나 보완한 핵심 설명
* 함께 수정한 README 또는 문서 파일
* 수행한 검증
* 검증하지 못한 항목
* 코드 동작 변경 여부

문서화되지 않은 기능이나 실행하지 않은 검증을 완료한 것처럼 보고하지 않는다.
